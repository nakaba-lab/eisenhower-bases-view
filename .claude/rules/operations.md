# 運用（リリース後の運用規律）規則

リリース後の運用 — **監視・障害対応（インシデント）・バックアップ/DR・データ移行・マニュアル・OSS ライセンス・秘密情報の管理** — の、言語・スタックに依存しない規律をここに定める。具体的なツール（監視 SaaS・バックアップジョブ・license ツール等）は固定資産として同梱せず、プロジェクト固有資産として `/project-setup`（`claude-automation-recommender`）が提案・蓄積する。

> **運用方針の合意（真実源）は `docs/要件定義書.md`**（「非機能要件」の監視方針・データ保護・バックアップ/DR、「リリース・運用」のデプロイ先・提供形態・データ移行。いずれも「標準／該当なし」の明示でも可）。本ファイルはその「標準」の中身＝各運用活動の定義・手順・記録の置き場を定める。

## 監視・オブザーバビリティ

- **方針（何を・どこまで監視するか）は `docs/要件定義書.md` の「非機能要件（監視方針）」で合意する**（「標準／該当なし」の明示でも可）。
- **具体導入**（エラートラッキング・ヘルスチェック・ログ整備・アラート）はテンプレートに固定資産を同梱せず、`/project-setup`・`/project-resync`（`claude-automation-recommender`）がスタックに合わせて提案・蓄積する。
- **導入した監視の設計**（何を監視し、どのアラートがどこへ届くか・しきい値の根拠）は `docs/design/operations.md` 領域に記録する（frontmatter は `.claude/rules/design-doc.md` に従う。`kind: operations`）。
- 監視は障害対応の入口: アラート・ヘルスチェックで検知した本番障害は incident として起票し（検知経路として記録）、下記の障害対応プロセスに乗せる。

## 障害対応（インシデント）プロセス

稼働中の本番に影響する障害は bug ではなく **incident テンプレート**（`.github/ISSUE_TEMPLATE/incident.md`／`.gitlab/issue_templates/incident.md`。ラベル/タイトル接頭辞 `incident`）で起票する（使い分けの定義は `.claude/rules/spec-driven.md`）。

### 重大度（severity）

| 重大度 | 定義 | 初動 |
|--------|------|------|
| **SEV1** | 全面停止（サービス全体が利用不能） | 他のすべての作業より優先して即応する |
| **SEV2** | 主要機能影響（主要機能が利用不能・著しく劣化） | 当日中に暫定対応へ着手する |
| **SEV3** | 軽微（限定的な影響・回避策あり） | トリアージのうえ通常の開発キュー（Milestone 配下）に乗せる |

**トリアージ**: インシデントの対応順は**重大度×影響範囲**（影響ユーザー・データ・期間）で決める。複数が同時に起きたら影響の大きいものから着手し、SEV1 の継続中は新規の機能開発を止めてよい。

> この重大度定義は **bug 票と共通**（bug のトリアージ基準〔重大度×頻度〕は `.claude/rules/spec-driven.md`）。「本番で起きているか」が incident と bug の分かれ目で、重大度の物差しは同じものを使う。

### 対応フロー

```
検知（監視アラート・ユーザー報告・定期点検 など）
  ↓
incident 起票（重大度・影響範囲・検知経路を記入。タイムラインは対応しながら追記する
              — 後からの再構成に頼らない）
  ↓
暫定対応（応急処置: 切り戻し・機能の一時停止 など、まず影響を止める。
        切り戻しの判断基準と手順は .claude/rules/git-workflow.md「ロールバック」）
  ↓
恒久対応（根本修正を fix Issue として起票し hotfix/* で実装〔.claude/rules/git-workflow.md〕。
        incident の「恒久対応」節から参照する）
  ↓
ポストモーテム（復旧後に下記様式で記入）→ incident クローズ
```

### ポストモーテム（様式）

復旧後、incident Issue の「ポストモーテム」節に記入する（記入完了がクローズ条件）。責任追及ではなく**原因究明と仕組み化**を目的とする。

| 項目 | 内容 |
|------|------|
| 根本原因 | 直接原因ではなく「**なぜ防げなかったか**」まで掘る（テスト・チェック・プロセスの欠落） |
| タイムライン | 検知〜復旧の時系列を確定させる（incident の「タイムライン」節） |
| 再発防止策 | 同種の障害を防ぐ具体策（テスト追加・バリデーション・チェック強化 など） |
| 仕組み化 | 再発防止策を強制力のある形に落とす: 回帰テスト・CI チェック・フック（`/hookify` で会話から恒久フックを生成できる） |

> **再発防止は意志ではなく仕組みで**。「気をつける」「注意する」は再発防止策ではない。忘れても機械（テスト・CI・フック）が止める形に落として初めてポストモーテム完了とする。

## バックアップ・リストア／災害復旧（DR）

- **RPO（目標復旧時点＝どこまでのデータ損失を許容するか）・RTO（目標復旧時間＝どれだけの停止を許容するか）は `docs/要件定義書.md` の「非機能要件（バックアップ/DR）」で合意する**（「標準／該当なし」の明示でも可）。
- バックアップは **取得周期（RPO が上限を決める）・世代（保持数・保持期間）・保管先（本番と障害を共有しない別系統）** の 3 点を決め、手順とともに `docs/運用ガイド.md`（下記「マニュアル体系」）に記す。
- **リストア演習**: バックアップの目的は「取れている」ことではなく「**戻せる**」こと。定期的に復元確認（リストア演習）を行い、手順が現状で通用するかと実測の復旧所要時間（≒RTO の実績値）を確認して `docs/運用ガイド.md` に記録する。演習なしのバックアップは無いものとして扱う。

## データ移行

提供開始・システム更改で既存データの移行が要る場合（有無は要件定義書「リリース・運用」で合意）、次の標準手順で進める:

```
移行設計（対象データ・新旧マッピング・クレンジング方針）
  ↓
リハーサル（本番相当のデータ量で投入手順・所要時間・検証手順を事前確認）
  ↓
本番切替（「何が起きたらどの時点へどう戻すか」＝切戻し条件を決めてから実施する）
  ↓
移行後検証（件数突合・整合性チェック・主要導線の疎通確認）
```

- 移行設計は **`docs/design/migration.md`** 領域（対象データ・マッピング・投入手順・リハーサル・切替計画。`.claude/rules/design-doc.md` の領域例）に記録する。
- リハーサルを飛ばして本番切替しない。切戻し条件が決まっていない切替も実施しない（戻れない移行は障害時に SEV1 へ直結する）。

## マニュアル体系

| マニュアル | 読者 | 内容 |
|-----------|------|------|
| `docs/操作マニュアル.md` | エンドユーザー | 主要機能の使い方（画面・操作手順・FAQ） |
| `docs/運用ガイド.md` | 運用者 | 起動/停止・バックアップ/リストア手順・設定変更・障害時の連絡先と初動 |

- **置き場は `docs/` 直下**: docs サイト有効時は Pages に自動公開され、目次（サイドバー）も自動更新される（`docs-site/README.md`。config 編集不要）。Pages の公開範囲はリポジトリの可視性と独立のため、公開が不適切な内容（障害時の連絡先・内部手順など）は `docs/` 外に置くか、可視性を確認のうえ載せる（OSS ライセンス節の「公開が不適切な納品物は配布物にのみ同梱する」と同じ逃がし）。
- **生成**: `/project-setup` が提供形態に応じて要否を確認のうえ雛形を生成する（エンドユーザー・運用者が居る提供形態で提案。library は通常不要）。
- **更新タイミング**: 操作・運用手順に影響するリリースで更新する。更新要否の確認は、リリース判定チェックリスト（`/release-notes`）と `/release-tasks` の「マニュアル・ドキュメント最終確認」タスクが担う（実装の都度ではなくリリース単位で同期を取る）。

## OSS ライセンス・第三者ソフトウェア一覧

- **依存追加時のライセンス確認はコードレビューの観点**（`.claude/rules/code-review.md`）: コピーレフト（GPL 系）の混入・商用利用可否を、依存を**追加した PR/MR の時点**で確認する（納品直前にまとめて発覚させない）。新規依存のライセンスと既知脆弱性は `security-reviewer` も確認する。
- **納品・配布時に第三者ソフトウェア一覧（使用 OSS とライセンスの一覧）が要る場合**は、手書きせずパッケージマネージャの依存一覧（lockfile）から生成する:

```bash
# 例（プロジェクトのスタックに読み替える。具体ツールの導入は /project-setup の recommender が提案する）
# npm:    npx license-checker --production --csv
# Python: pip-licenses --format=markdown
# Java:   Gradle/Maven の license レポートプラグイン
# Rust:   cargo license
```

- 生成した一覧は納品要件に応じて `docs/` 配下に置く（Pages に公開される。公開が不適切な納品物は配布物にのみ同梱する）。

## 秘密情報の管理

`.env`・API キー・トークン・接続文字列などの秘密情報は、**役割の異なる多段ガード**で漏えいを防ぐ（設計書・テストの多段ガードと同じ思想 — 1 層のすり抜けを次の層が捕捉する。単層に依存しない）:

| 層 | 仕組み | 守る範囲・限界 |
|----|--------|--------------|
| コミット対象からの除外 | `.gitignore`（`.env`／`.env.*` を除外。`!.env.example` で設定例のみ track） | 実秘密ファイルの誤 track を防ぐ（リネームされたファイル・ソース内ハードコードには無力） |
| セッション層 | `security-guidance` プラグイン（出荷時有効） | Claude 経由の編集・コミット時に hardcoded secrets を警告（best-effort。人間の直接コミット・Claude を介さず置かれたファイルには届かない） |
| push 保護（ホスト機能） | GitHub: Secret scanning ＋ Push protection（public 無料・private は有償アドオン）／GitLab: Secret push protection は Ultimate のみ | push の瞬間にホスト側で止める（`--no-verify` でもすり抜け不可）。有効化は `/project-setup` の「バージョン管理・リモート同期」タスクが案内する（プラン依存のため案内に留める） |
| CI スキャン | 同梱の `secret-scan`（gitleaks・MIT。`.github/workflows/secret-scan.yml`／`.gitlab/ci/secret-scan.yml`）。GitLab はホストの Secret Detection テンプレート（include・全ティア無料）も併用できる | PR/MR の差分コミットを決定論的にスキャンし、検出は fail（gitleaks の取得失敗等の環境問題は警告して成功＝CI 自体を壊さない） |

- **スキャン範囲は PR/MR の差分コミットのみ**（既存リポジトリへ後から導入した時に、過去履歴の検出で CI が埋まるのを避ける）。既存リポジトリの初回は全履歴を一度点検する — GitHub は `secret-scan` の手動実行（workflow_dispatch＝全履歴モード）、GitLab はローカルで `gitleaks git`（引数なし＝全履歴）を実行する。
- **逃がし（誤検知）**: テスト用ダミー鍵などの誤検知は、当該行末に **`gitleaks:allow` インラインコメント**を付けるか、リポジトリ直下の **`.gitleaks.toml`**（allowlist）でパス・ルール単位で除外する（gitleaks ネイティブの仕組み。除外は設定としてリポジトリに残るためレビューで追える）。
- **検出されたら（漏えい時の原則）**: push された秘密は**漏えいしたものとみなし、必ず失効・ローテーション**する（履歴の書き換え・コミットの削除だけでは不十分 — clone・キャッシュ・フォークに残る）。本番影響があれば incident として起票し、上記の障害対応プロセスに乗せる。
- **`docs/` に秘密を置かない**: `docs/` 全体は Pages へ自動公開される（`docs-site/README.md`）。接続文字列・内部 URL・実鍵を含む手順は `docs/` に書かず、保管場所（シークレットマネージャ等）への参照に留める（「マニュアル体系」節の公開可否の確認と対）。

## ロールバック

リリース・マージの切り戻し（前進修正 `hotfix/*` と `git revert` の使い分け・squash merge コミットの revert 手順・リリースタグの扱い）の正は **`.claude/rules/git-workflow.md` の「ロールバック」節**（本ファイルでは重複定義しない）。インシデントの暫定対応として切り戻す場合もそちらに従う。
