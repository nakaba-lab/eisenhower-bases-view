# 要件ディスカッション（論点すり合わせ・理解の照返し）

> 対象フロー: 新規のみ ｜ 種別: 【人間ゲート】

> ヒアリング（「ヒアリング充足ループ」タスク）が「聞いた回答で項目を埋める」工程なのに対し、ここは**議論して認識を合わせる**工程。項目が埋まっていても、利用者自身が気づいていない論点・暗黙の前提・解釈違いは潰れていないため、要件定義書を書く前に必ず通す。**作法の正は `.claude/rules/alignment.md`**（本タスクは適用箇所と完了条件のみを定め、手順の詳細を重複定義しない）。

1. **論点ディスカッション**：「ヒアリング充足ループ」タスクの聴取内容から論点・盲点を抽出する（観点: エッジケース／データ整合性／権限・ロールの粒度／運用負荷／性能・スケール前提／既存資産との整合／**利用者が言及していないが決めるべきこと**）。抽出した論点は **1 論点ずつ `AskUserQuestion`** で「選択肢＋トレードオフ＋推奨」を提示して確定する（複数論点をまとめて列挙して流さない）。議論しても決着しない論点は無理に確定せず「**未決事項**」として控える（「要件定義書の作成」タスクで要件定義書の「未決事項」節に記録する）。
2. **理解の照返し（playback）**：節の見出し列挙ではなく「**私はこう理解した**」の構造化サマリ — 目的／スコープ境界（**やらないこと**を含む）／主要フローの自分の言葉での再話／前提 — を照返し、利用者が**認識一致を明示するまで**修正を反復する（回数上限なし）。「ヒアリング充足ループ」タスクの前進保証でデフォルト適用した項目は、必ず「未決事項」として照返しに含める（黙って確定しない）。
3. **省略の自己判断をしない（常に必須）**：「議論すべき論点が無い」と判断した場合も、その判断と根拠（聴取内容の要約・論点が無いと考えた理由）を `AskUserQuestion` で利用者に照返し、確認を得てから次へ進む（no-op にも人間の承認が要る。`.claude/rules/alignment.md` の大原則）。

`【完了条件】抽出した全論点が「確定」または「未決事項に記録」のいずれかになり、理解の照返しに対して利用者が認識一致を明示した（論点なしと判断した場合は、その判断への AskUserQuestion の確認回答）。【証拠なしに completed 禁止】認識一致（または確認回答）を得る前に completed にしない。`
