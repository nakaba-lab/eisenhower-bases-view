# 要件定義書の作成

> 対象フロー: 新規のみ ｜ 種別: 【人間ゲート】

**作成直前の最終ゲート**：要件定義書を書く前に「ヒアリング充足ループ」タスクの必須チェックリストを再判定する。未充足の必須項目が残っていて、かつ利用者が（全体ゲートで）まだ脱出していなければ、**「ヒアリング充足ループ」タスクの深掘りループへ戻す**。ただし戻り先で**停滞**している必須項目は同タスクの前進保証（停滞 → デフォルト適用）を必ず経由させ、「戻るが進めない」閉路に入れない。充足済み、または利用者が脱出済みなら次へ進む。

ヒアリング内容と「要件ディスカッション」タスクの確定事項をまとめ、ユーザー承認後に `docs/要件定義書.md`（プロジェクト概要書）を作成する。必須節は次の **10 節（提供形態が Web の場合は「UI/UX 方針」を含む 11 節）**。ヒアリングで「標準／該当なし」と確定した項目は、その旨を節内に明記して埋める（節自体は省略しない）：

1. **背景・目的** — 解決する課題・ゴールに加え、**想定ユーザー／アクター（権限ロール含む）**を必ず書く（Web に限らず CLI・library でも「誰が・どの権限で使うか」を定義する）。
2. **スコープ**
3. **主要機能**
4. **業務フロー** — 主要ユースケースの流れ（アクターと操作の順序）を **Mermaid flowchart／sequenceDiagram** のコードブロックで図示する。業務システム・Web では必須、それ以外は「該当なし」の明示でも可。
5. **非機能要件** — 対象環境・セキュリティに加えて、**定量目標**（応答時間・同時利用者数・データ規模）／**データ保護**（個人情報の有無・保持/削除・暗号化）／**バックアップ/DR**（RPO/RTO）／**監視方針**／（深掘り観点）**i18n・タイムゾーン**。いずれも「標準／該当なし」の明示でも可。
6. **テスト方針** — どのテストレベル（単体／結合／総合／受入／性能）をどこまで実施するかの合意（体系は `.claude/rules/testing-strategy.md`）。「標準＝TDD（単体）＋結合は分離」の明示でも可。受託・業務システムでは、納品物としてのテスト仕様書（`docs/test/`）と受入確認（UAT）の要否も合意する（テスト仕様書の規律は `.claude/rules/testing-strategy.md`）。
7. **技術選定**
8. **リリース・運用** — デプロイ先・提供形態（配布方法）・**データ移行**の有無（「該当なし」の明示でも可）。ここの確定値は「プロファイル確定」タスクの `commands.deploy` とデプロイ CI 雛形の有効化判断に使う。
9. **未決事項** — 決めきれなかった論点・デフォルト適用した項目・先送りの理由と確認予定の一覧（**表: 項目／現在の仮置き／確認先・時期**）。「要件ディスカッション」タスクで決着しなかった論点と、ヒアリング・照返しでデフォルト適用した項目をここに集約する（無ければ「なし」を明示。節自体は省略しない）。解消したら本文の該当節を改訂して「変更履歴」に記録し、この表から消す（管理の作法は `.claude/rules/alignment.md`「未決事項の管理」）。
10. **変更履歴** — 日付・変更内容・合意者の表。**初版の 1 行を必ず入れる**。以後、要件が変わったら該当節を改訂してこの表に追記する（現状との乖離は `/project-resync` の整合点検が検出する）。
11. **（Web のみ）UI/UX 方針** — 「ヒアリング充足ループ」タスクの UI 要求から: ターゲットユーザーと利用文脈／デザインの方向性・参照デザイン／主要画面の構成／デザインシステム・コンポーネントライブラリの採否／既存ブランド資産。この節は Issue の視覚 AC（`spec-driven.md`）と `frontend-reviewer` のデザイン意図整合チェックの照合元になる。

**draft は照返しとともに提示する（認識一致まで反復）**：draft の提示は節の見出し列挙ではなく、「私はこう理解した」の構造化サマリ（「要件ディスカッション」タスクと同じ作法＝`.claude/rules/alignment.md`）を添えて行い、利用者が認識一致を明示するまで修正を反復する（回数上限なし）。
**脱出・デフォルト化で確定した項目は、採用したデフォルトを「前提」として節内に明記し、あわせて「未決事項」節にも記録して照返しに含める**（黙って確定しない）。docs サイト有効時はこのファイルも公開ページとして描画されるため、先頭に `title: 要件定義書`（任意で `description:`）の frontmatter を付ける（欠けてもファイル名で補完されてビルドは落ちないが、明示するとナビ・タブ・メタが安定する）。

`【完了条件】docs/要件定義書.md のパスと必須 10 節（Web 時は「UI/UX 方針」を含む 11 節）の見出しを提示し、照返しへの認識一致を経て利用者が内容を承認した。【証拠なしに completed 禁止】ファイルを実際に作成しパスを提示するまで completed にしない（「作成予定」は完了ではない）。`
