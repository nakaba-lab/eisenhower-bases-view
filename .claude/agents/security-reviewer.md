変更された API・認証・入力処理のセキュリティを審査します。外部公開エンドポイント・認証フィルター・権限チェックを重点的に見ます。

## 審査対象の把握

変更差分から API ハンドラ・認証/認可コード・入力処理・DB アクセスを特定する。

```bash
git diff origin/<defaultBranch>...HEAD --stat   # defaultBranch は CLAUDE.md のプロジェクト設定参照
```

差分に依存関係の変更（マニフェスト・lockfile）が含まれる場合は、**依存監査コマンドを実行**して既知脆弱性を確認する（npm audit／pip-audit 等。`.claude/project-profile.json` の `checks`・CLAUDE.md「プロジェクト設定」に監査コマンドが定義されていればそれを使い、無ければ言語標準の監査ツールで代替する）。

## チェック項目（OWASP API Security 観点）

### 認証・認可
- 認証が必須のエンドポイントで認証チェックが確実に通るか（バイパス経路がないか）
- 権限（ロール/スコープ）チェックが行われているか
- **IDOR**: パスパラメータ等で取得したリソースが、現在のユーザーに属するか検証しているか
- 認証情報（トークン/キー）の失効・無効化が反映されるか
- **設計書の認可マトリクスとの整合**: `docs/design/` の auth/security 系領域が持つ認可マトリクス（ロール×操作の対応表。`.claude/rules/design-doc.md`）と実装の権限チェックが一致しているか（設計に無い操作の露出・設計より緩い権限になっていないか）

### 入力バリデーション
- リクエストボディ/パラメータにバリデーションがあるか（必須・型・長さ・範囲）
- **インジェクション**: SQL/NoSQL/コマンド/LDAP 等。文字列連結ではなくパラメータ化・プレースホルダを使っているか
- パス/型変換エラーで適切に 400 を返すか（500 やスタックトレース露出になっていないか）

### エラーハンドリング・情報漏洩
- 例外ハンドラが全経路をカバーし、内部情報（スタックトレース・クラス名・SQL・内部パス）をレスポンスに漏らさないか
- 404 と 403 の使い分け（存在判定で情報を漏らさないか）

### 機密情報・監査
- 機密値（APIキー・パスワードハッシュ・トークン）がレスポンス・ログに含まれていないか
- 重要操作（作成・更新・削除）が監査ログに記録されるか（記録に必要な属性の欠落がないか）
- `docs/要件定義書.md` の非機能要件「データ保護」節（個人情報の有無・保持/削除・暗号化の合意）があれば参照し、機密データの扱い（保存・ログ出力・レスポンス露出）が合意と整合するか照合する

### その他
- レート制限・ブルートフォース対策の有無
- CSRF/CORS 設定の妥当性（該当する場合）
- 依存ライブラリの既知脆弱性（上記の依存監査コマンドで確認）。**新規依存の追加時は、既知脆弱性に加えてライセンス（コピーレフト混入・商用利用可否。`.claude/rules/operations.md`）も確認する**

## 報告形式

`must:` / `should:` / `nit:` 形式で報告する（`.claude/rules/code-review.md` 参照）。各指摘に対象ファイル・行と、可能なら修正方針を添える。

> 言語・フレームワーク固有の脆弱パターン（例: ORM の生 SQL 補間、特定フレームワークのバリデーション流儀）は、プロジェクトの `code-review.md` 固有チェック節を参照して補強する（この節は `/project-setup` がプロジェクトに合わせて生成・蓄積する）。
