# 要件定義書の有無確認

> 対象フロー: 既存のみ ｜ 種別: 【人間ゲート】

既存プロジェクトはヒアリング充足ループ・要件ディスカッション・要件定義書の作成（いずれも新規のみ）の各タスクを通らない。要件定義書はテスト方針・非機能の定量目標・要件→Issue のトレース（`.claude/rules/spec-driven.md`）の照合元になるため、ここで有無を確認し、無ければ**軽量に補う**か **opt-out を明示記録**する（どちらも無いと、後続の要件参照系が「未確認」のまま無言で素通りする）。

1. **存在確認**：`docs/要件定義書.md`（または同等の要件文書。あればパスを確認して読み替える）が存在するか確認する。存在すれば、必須節（「要件定義書の作成」タスクの節構成）に大きな欠落が無いかを一瞥して報告し、完了する（補筆はここでは強制しない＝以後の乖離・欠落は `/project-resync` の整合点検が継続的に担う）。
2. **無い場合（`AskUserQuestion`）**：
   - header: `要件定義書`
   - question:「このプロジェクトには `docs/要件定義書.md` がありません。テスト方針・非機能目標・要件→Issue のトレースの照合元になるため、`/init` のコードベース要約を種にした軽量ヒアリングでの作成を推奨します。どうしますか？」
   - options（**multiSelect は使わない**）:「軽量ヒアリングで作成する（推奨）／作成しない（opt-out を CLAUDE.md に記録）」
3. **作成する場合（軽量ヒアリング＋論点ディスカッション（軽量版）・理解の照返し）**：`/init` が CLAUDE.md に追記したコードベース要約（言語・構成・主要コマンド）から読み取れる節（主要機能・技術選定等）は**推定で埋めて確認に回し**、読み取れない節だけを `AskUserQuestion`（1 回最大 4 問）で聴取する：**目的／スコープ／非機能要件（定量目標・データ保護・バックアップ/DR・監視。「標準／該当なし」明示可）／テスト方針／（Web 相当のみ）UI/UX 方針**。節構成は新規フローの「要件定義書の作成」タスクと同じ 10 節（Web 時 11 節。業務フロー・リリース・運用は「該当なし」の明示でも可、決めきれなかった論点・デフォルト適用した項目は「未決事項」節に表で記録、「変更履歴」節に初版 1 行）。生成の前に**論点ディスカッションの軽量版と理解の照返し**を行う（作法の正は `.claude/rules/alignment.md`）：聴取・推定の内容から決めるべき論点を抽出して `AskUserQuestion` で確定し（決着しない論点は「未決事項」節へ）、「私はこう理解した」の構造化サマリを照返して認識一致を得てから生成する（論点が無いと判断した場合も、その判断を確認してから進む）。新規フローの充足ループほどの反復はせず **1〜2 ラウンドで前進**させる（埋まらない節は「標準／該当なし」のデフォルトを適用して明示し、「未決事項」節にも記録する）。承認を得て `docs/要件定義書.md` を生成する（先頭に `title: 要件定義書` の frontmatter）。
4. **作成しない場合（opt-out）**：CLAUDE.md のプロジェクト固有追記欄（コーディング規約節の下の追記領域）に「要件定義書は持たない（理由・判断日）」を 1 行記録する（後続の要件参照系〔planning のトレーサビリティ・`/project-resync` の整合点検〕が「対象外」と判定できる根拠になる）。

`【完了条件】既存の要件定義書のパス確認、生成した docs/要件定義書.md のパス＋承認、または opt-out の CLAUDE.md 記録差分、のいずれかを提示。【証拠なしに completed 禁止】AskUserQuestion の回答・承認を得る前に completed にしない。`
