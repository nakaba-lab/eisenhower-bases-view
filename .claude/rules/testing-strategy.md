# テスト戦略（テストレベル体系）規則

単体テスト（`.claude/rules/tdd.md`）より上のテストレベル — **結合・総合（システム）・受入（UAT）・性能** — の体系をここに定める。言語・テストフレームワークに依存しない原則だけを書き、具体的なコマンドは **CLAUDE.md の「プロジェクト設定」節**（`commands.test`／`commands.integrationTest`／`commands.coverage`）を参照する。

> **テスト戦略の合意（真実源）は `docs/要件定義書.md` の「テスト方針」節**。どのレベルをどこまでやるかは `/project-setup` のヒアリングで合意して同節に残す（「標準＝TDD＋結合は分離」の明示でも可）。本ファイルはその「標準」の中身＝各レベルの定義・実施タイミング・記録の置き場を定める。

## テストレベル体系（一覧）

| レベル | 検証対象 | 基準（何に照らすか） | 実行方法 | 実施タイミング |
|--------|---------|--------------------|---------|--------------|
| 単体 | 関数・クラス・コンポーネント単位のロジック | L2/L3 Issue の AC（Given/When/Then） | `commands.test` | 常時（Red-Green-Refactor。`.claude/rules/tdd.md`） |
| 結合 | 複数コンポーネントの連携・実 DB/外部 I/O | L2 Issue の AC のうち結合を要するもの | `commands.integrationTest` | L2 Issue の全 AC 達成時＋コミット前ゲート |
| 総合（システム） | システム全体（実環境相当で通し） | 要件定義書の主要機能・非機能（定量目標）＋ L1 Milestone の AC | 全テストスイート＋（Web 時）playwright E2E | リリース前（`/release-tasks` がタスク敷設） |
| 受入（UAT） | 利用者/発注者による確認 | L1 AC の受入チェックリスト | 人間（利用者/発注者）の確認 | リリース前（受託・業務システムの場合のみ） |
| 性能 | 応答時間・同時利用者数・データ規模 | 要件定義書の非機能（定量目標） | プロジェクト固有（recommender が提案） | 定量目標がある場合、総合テストと合わせて |

### 単体テスト

AC 駆動の TDD（Red-Green-Refactor）で書く。命名・AC→テスト変換・原則は `.claude/rules/tdd.md` が正（本ファイルでは重複定義しない）。

### 結合テスト

- **対象**: 複数コンポーネントの連携と、実 DB・実外部 I/O を伴う検証。`.claude/rules/tdd.md` の「ユニットテストは外部 I/O に依存させない」原則で単体から分離した重い統合（同ファイルの言う「統合テスト」）がこのレベルに当たる。
- **配置規約**: 単体テストと**実行単位を分離**できる形で置く（ディレクトリ分割・タグ/マーカー・命名 suffix 等、プロジェクトのテストランナーの流儀に従う）。`commands.integrationTest` 単独で結合テストだけを実行できる状態を保つ。
- **実行**: `commands.integrationTest`（CLAUDE.md「プロジェクト設定」）。**未設定なら「結合テストは単体に統合されている」扱い**でよい（`commands.test` がすべてを実行する）。
- **実施タイミング**: L2 Issue の全 AC 達成時（実装 Green の確認）と、コミット前ゲート（`/dev-tasks` の「コミット」タスクが設定時のみ verification に含める）。

### 総合テスト（システムテスト）

- **対象**: リリース対象のシステム全体。実環境相当（ステージング等、本番に近い構成）で通しで検証する。
- **基準**: `docs/要件定義書.md` の主要機能・非機能要件（定量目標）と、リリース対象 L1 Milestone の AC。
- **Web プロジェクトは `playwright` で主要ユースケースの E2E**（画面遷移・主要導線の通し操作）を含める。
- **実施**: リリース工程の一部として `/release-tasks` がタスク敷設する。記録は `docs/test/` に残す（下記「計画と記録」）。

### 受入テスト（UAT）

- **対象**: 受託開発・業務システムなど、利用者/発注者による受入確認が要るプロジェクトのみ（該当しなければ実施不要を明示してよい）。
- L1 Milestone の AC を**受入チェックリスト**に変換し、利用者/発注者が確認する（人間ゲート。`/release-tasks` の受入チェックリストタスクが敷設する）。
- L1 Milestone の DoD「（受託・業務システムの場合）利用者/発注者による受入確認（UAT）が完了している」（`.claude/rules/spec-driven.md`）に対応する。

### 性能テスト

- **対象**: 要件定義書の非機能要件に**定量目標**（応答時間・同時利用者数・データ規模等）がある場合のみ。
- **計測方法と合否基準**を定量目標から決める（目標値＝合否ライン。計測条件〔データ量・並列数〕も記録に残す）。性能/負荷テストランナーはプロジェクト固有資産として `/project-setup`（`claude-automation-recommender`）が提案・蓄積する。
- **実施**はリリース前（総合テストと合わせて）。結果は `docs/test/` の総合テスト記録に含める。

## テストデータ・環境管理

- **フィクスチャ/シードの置き場をプロジェクトで定め、再現性を原則とする**: テストは何度実行しても同じ結果になるよう、外部状態に依存しない初期化・後片付けを持つ（共有環境の残骸に依存するテストを作らない）。
- **個人情報を含む本番データを直接テストに使わない**。必要ならマスキング（匿名化）したデータを使う（`docs/要件定義書.md` の非機能要件「データ保護」の合意に従う）。
- テスト DB・コンテナ等の環境定義はプロジェクト固有資産として `/project-setup`（`claude-automation-recommender`）が生成・蓄積する（テンプレートは固定資産を同梱しない）。
- ステージング相当（本番に近い構成）での確認は**総合テスト**で行う（単体・結合ではスタブ/ローカル環境で十分）。

## テスト仕様書（テスト設計・受託で必須）

AC から導くテストケース・シナリオを書面化したテスト設計成果物。TDD（`.claude/rules/tdd.md`）は AC→テストコードへ直接変換するが、**受託・業務システムでは納品物としてテスト仕様書（テスト項目表）が要求されることが多い**ため、書面のテスト設計層を成果物として持つ。

- **置き場**: `docs/test/<機能>-test-spec.md`（総合テストの実行記録と同じ `docs/test/`）。
- **内容**: テストケース一覧（番号・前提・操作・期待結果・合否欄）／テスト観点（正常系・異常系・境界値・権限）／AC との対応（どの AC を満たすか）。
- **要否**: **受託・業務システムでは必須**（`docs/要件定義書.md`「テスト方針」で合意。`.claude/rules/spec-driven.md` の L1 Milestone DoD に条件付き項目として乗り、finalize のクローズ前ゲート〔UAT と同じ人間ゲート〕で担保される）。内製で不要なら「テスト方針」節に不要を明示する。
- **公開注意**: `docs/` 配下は Pages に自動公開されるため、内部テストデータ（実データ・接続情報・秘密）は載せず参照に留める（`.claude/rules/operations.md`「秘密情報の管理」）。

## 計画と記録

| 成果物 | 置き場 | 内容 |
|--------|--------|------|
| テスト戦略の合意 | `docs/要件定義書.md` の「テスト方針」節 | どのレベルをどこまでやるか（「標準＝TDD＋結合は分離」の明示でも可） |
| テスト仕様書（テスト設計・受託で必須） | `docs/test/`（例: `docs/test/<機能>-test-spec.md`） | AC から導くテストケース一覧・観点・境界値と AC との対応。**受託・業務システムでは必須**（要件定義書「テスト方針」で合意・上記「テスト仕様書」節）。`docs/` 配下＝Pages 公開のため内部テストデータは載せない |
| 総合テスト（システムテスト）の実行記録 | `docs/test/`（例: `docs/test/release-v1.2.0-system-test.md`） | 実行日時・対象（バージョン/コミット）・結果・証跡（スクリーンショット・ログ）。`docs/` 配下なので Pages に自動公開される |
| 機械記録 | CI のチェックラン | 単体・結合（CI に組み込んだ分）の実行結果は CI が自動で記録する（手書きの転記はしない） |

- **カバレッジ**: `commands.coverage`（CLAUDE.md「プロジェクト設定」）が設定されている場合、コードレビュー時に計測し、**著しい低下**があれば報告する（`/dev-tasks` の「コードレビュー」タスク・`.claude/rules/code-review.md`）。数値目標は強制しない（カバレッジは劣化検知のシグナルであって品質の証明ではない）。

## 関連

- 単体テスト（Red-Green-Refactor・AC→テスト変換・命名）は `.claude/rules/tdd.md`。
- リリース工程（総合テスト→受入チェック→タグ→リリース後スモーク）のタスク敷設は `/release-tasks` スキル。
- AC/DoD の 3 階層構造（L1/L2/L3）は `.claude/rules/spec-driven.md`。
