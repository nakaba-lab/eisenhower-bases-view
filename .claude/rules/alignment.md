# すり合わせ（アラインメント）規則

実装前のすり合わせ — **要件定義・基本設計・詳細仕様の認識合わせ** — の作法（曖昧さ規律・論点ディスカッション・理解の照返し・設計オプション比較・AC ウォークスルー・未決事項管理）の真実源をここに定める。各スキル（`/project-setup`・`/github-planning`・`/gitlab-planning`・`/dev-tasks`）と他の規則ファイルは本ファイルを参照し、作法を重複定義しない。

## 基本概念

- **AI 駆動開発では、実装前のすり合わせが品質を決める。** 実装は速く反復できるが、要件・設計の誤解は実装の速さでは直らず、「正しく動く間違ったもの」として大量に現れる。コードを書く前の認識合わせの厚さが、そのまま成果物の質になる。
- **「埋める」と「議論する」を区別する。** 項目を埋める（ヒアリングの回答を書き写し、テンプレートの節を充足させる）だけでは、利用者自身が気づいていない論点・暗黙の前提・解釈違いが素通りする。すり合わせとは、AI が論点を提起し、選択肢とトレードオフを示し、利用者と**議論して**認識を合わせる活動である。

### 大原則: 常に必須（省略の判断も人間が行う）

**すり合わせ系ステップの省略（no-op）をエージェントの自己判断で行わない。**「設計に影響しない」「議論すべき論点が無い」と判断した場合も、その判断と根拠（変更予定の範囲・論点が無いと考えた理由）を `AskUserQuestion` で利用者に照返し、**確認を得てから**先へ進む（= no-op にも人間の承認が要る）。

> **証拠による対象外判定とは別物**: `/dev-tasks` の「ビジュアル/UX 確認」のように、機械的な証拠（差分に UI ファイルの変更が 0 件・`frontendDir: "none"`）で対象外を判定する no-op は本原則の対象ではない（あちらは証拠の提示が要件）。本原則が対象とするのは、エージェントの**解釈・判断**によって省略しうるすり合わせ系ステップ。
>
> **自律実行（`/goal`）と矛盾しない**: すり合わせの作法はいずれも `AskUserQuestion`（人間の入力を待つ構造的な停止）を内包するため、自律周回はそこで必ず止まる。加えて `/dev-tasks` では設計のすり合わせ（「実装前設計」）を `/goal` の自律周回の**前段**に置き、設計承認を得てから自律区間（テスト先行〜レビュー）を促す（設計はやり直しコストが最大のフェーズなので、自律モードに入った後の片手間の確認に委ねない）。実装前設計は auto（人間の手動実行を待たず Claude が `AskUserQuestion` で承認を取りながら進める＝【自律対象外】ではない）だが、`/goal` 周回には含めず**その前段で同期完了させる**という位置づけ。これにより自律区間（テスト先行〜コミット）の証拠規律とも衝突しない。

> **AC/DoD の「完了」は助言的な歯止め＋人間ゲートで閉じる（機械写像ではない）**: 完了の証拠を機械で全数突合する閉ループは**意図的に持たない**＝人間ゲートに集約するトレードオフ。`pre_task_update` フックは completed の瞬間に証拠提示を促すリマインダーを出すが**非ブロック（常に exit 0・許可フローを変えない）**で、AC 項目と完了証拠を機械的に対応づける写像も持たない（AC 採番のトレースをしない方針＝`.claude/rules/spec-driven.md`）。最終的な担保は finalize スキルのクローズ前ゲート（本文 `[x]` 確認・未充足は `AskUserQuestion`）＝**人間ゲート**に置く。ゆえに完了ループを完全自動で閉じる方向の機械強制（completed のブロック化等）は採らない（証拠ベース完了はあくまで人間が確認する設計）。

## 5 つの作法

| 作法 | フェーズ | 主な実施場所 |
|------|---------|-------------|
| 曖昧さ規律（推測で埋めない） | 横断（全工程） | すべてのスキル・タスク |
| 論点ディスカッション | 要件定義 | `/project-setup`「要件ディスカッション」（既存導入では「要件定義書の有無確認」の軽量版） |
| 理解の照返し（playback） | 要件定義 | `/project-setup`「要件ディスカッション」「要件定義書の作成」（既存導入では「要件定義書の有無確認」内） |
| 設計オプション比較 | 基本設計 | `/dev-tasks`「実装前設計」 |
| AC ウォークスルー | 詳細仕様 | `/github-planning`・`/gitlab-planning` の起票前ゲート、`/dev-tasks`「実装前設計」 |

### 1. 曖昧さ規律（推測で埋めない）

横断原則。**仕様の曖昧さ — 複数解釈可能な語・未定義の境界値・暗黙の前提・矛盾 — を検出したら、推測で埋めずに実装前に `AskUserQuestion` で確認する。**

- 確認結果は **Issue 本文（AC の修正）** または **設計書の「主要な設計判断」節**（`.claude/rules/design-doc.md`）に記録し、会話の中だけに残さない（チャットログは記録先ではない）。
- 確認するほどでない微細な判断は、**採った解釈を成果物（設計書・Issue 本文・PR/MR 説明等）に明記する**（無言の解釈をしない）。
- 要件定義・設計・起票だけでなく**実装中も適用する**（`/dev-tasks` の「実装（Green）」で曖昧さに気づいたら、推測で実装する前に確認する）。

### 2. 論点ディスカッション（要件定義）

ヒアリング充足ループの**後**・要件定義書作成の**前**に行う（`/project-setup` の「要件ディスカッション」タスク。既存プロジェクト導入で要件定義書を生成する場合は「要件定義書の有無確認」タスク内の軽量版）。

- AI が聴取内容から**論点・盲点を抽出**する。観点: エッジケース／データ整合性／権限・ロールの粒度／運用負荷／性能・スケール前提／既存資産との整合／**利用者が言及していないが決めるべきこと**。
- **1 論点ずつ** `AskUserQuestion` で**選択肢＋トレードオフ＋推奨**を提示して確定する（まとめて列挙して「問題ないですね」で流さない）。
- 決着しない論点は「未決事項」へ記録する（下記「未決事項の管理」）。
- 論点が無いと判断した場合も、その判断を照返して確認を得る（上記「大原則」）。

### 3. 理解の照返し（playback）

要件定義書 draft の提示時、節の見出し列挙ではなく**「私はこう理解した」の構造化サマリ**を照返す:

- **目的**（何のために作るか）
- **スコープ境界**（やること・**やらないこと**）
- **主要フローの自分の言葉での再話**（聴取の復唱ではなく、AI の理解で語り直す）
- **前提**（成り立っていると仮定していること）

利用者が**認識一致を明示するまで**修正を反復する（回数上限なし）。前進保証（停滞時のデフォルト適用＝`/project-setup` のヒアリング充足ループ）は維持するが、**適用した項目は必ず「未決事項」に記録し、照返しに含める**（黙って確定しない）。

### 4. 設計オプション比較（基本設計）

`/dev-tasks` の「実装前設計」で設計判断を伴う場合、draft を書く**前に 2〜3 案＋トレードオフ表**を示して選択を得る:

- 比較軸: **複雑さ・拡張性・性能・工数・既存との整合**。
- `AskUserQuestion` で選択を得る（案の構造・トレードオフ表は preview で提示してよい）。
- **1 案しか妥当でないと判断した場合も「なぜ代替が無いか」を添えて確認を得る**（上記「大原則」＝比較の省略も人間が判断する）。
- **UI を含む Issue はワイヤーフレーム/レイアウトを 2 案以上比較**する（Mermaid flowchart か ASCII）。
- 選択の理由は設計書の「主要な設計判断」節に記録する（`.claude/rules/design-doc.md`）。

### 5. AC ウォークスルー（詳細仕様）

Given/When/Then を**平文の利用シナリオに展開して照返し**、解釈違い（境界値・状態の定義・暗黙の前提）を潰す。

```
AC: - [ ] Given 無効な API キーを持つユーザー, When 一覧 API を呼ぶ, Then 401 が返る

シナリオ展開: 「期限切れのキーを持つ利用者が一覧画面を開くと、データは表示されず
             再ログインを促すエラーになります。『無効』には期限切れ・失効・形式不正の
             3 つを含む想定ですが、この理解で合っていますか？」
```

実施場所は 2 つ（作法の正は本ファイル）:

- **planning（`/github-planning`・`/gitlab-planning`）**: 起票前ゲート（人間承認）の強化として、**L2 Issue すべて**について AC をシナリオ展開して提示し、確認を得てから起票する。要件定義書の「未決事項」に依存する Issue は**起票前に停止してその論点を確定する**（または Issue 本文に未決依存を明記する）。
- **`/dev-tasks`「実装前設計」**: 着手時に対象 Issue の AC をシナリオ照返しする。planning でウォークスルー済みなら**「済みであること」を利用者に確認するだけでよい**（二重の全展開はしない）。

## 未決事項の管理

決めきれなかった論点・デフォルト適用した項目を「黙った先送り」にせず可視化する装置。`docs/要件定義書.md` の必須節「**未決事項**」（必須 10 節・Web 時 11 節の 1 つ。節構成は `/project-setup` が正、見出しの存在は CI〔`design-doc-check` の `requirements-doc-check` ジョブ〕も点検する）に表で残す:

| 項目 | 現在の仮置き | 確認先・時期 |
|------|------------|-------------|
| <決めきれなかった論点・デフォルト適用した項目> | <いま採用している暫定値・先送りの理由> | <誰に・いつ確認するか> |

- **記録するもの**: 論点ディスカッションで決着しなかった論点／前進保証で適用したデフォルト／理解の照返しで保留になった前提。
- **参照するもの**: planning は起票前に未決事項を確認し、依存する Issue を起票前に確定するか依存を明記する（上記「AC ウォークスルー」）。
- **解消したら**: 本文の該当節を改訂して「変更履歴」節に記録し、未決事項の表から消す（未決事項は溜め込む場所ではなく、解消されて空に向かうリスト）。

## 記録先の対応表

すり合わせの結果は会話の中だけに残さず、必ず成果物に記録する:

| すり合わせの結果 | 記録先 |
|----------------|--------|
| 確定した設計判断（設計オプション比較の選択理由・曖昧さ確認の結果） | `docs/design/<領域>.md` の「主要な設計判断」節（`.claude/rules/design-doc.md`） |
| 確定した要件の変更（論点ディスカッション・照返しで決まったこと） | `docs/要件定義書.md` の該当節を改訂＋「変更履歴」節に追記 |
| 決めきれなかった論点・デフォルト適用した項目 | `docs/要件定義書.md` の「未決事項」節 |
| AC の解釈確定（ウォークスルー・曖昧さ確認の結果） | Issue 本文（AC の修正・補記） |

## 適用箇所の対応表

どのスキルのどのタスク・工程がどの作法を使うか（タスクの手順詳細は各スキルが持ち、作法の中身は本ファイルが正）:

| スキル・タスク | 使う作法 |
|---------------|---------|
| `/project-setup`（新規）「要件ディスカッション（論点すり合わせ・理解の照返し）」 | 論点ディスカッション＋理解の照返し |
| `/project-setup`（新規）「要件定義書の作成」 | 理解の照返し（draft の照返し）＋未決事項節の生成 |
| `/project-setup`（既存）「要件定義書の有無確認」 | 生成する場合のみ: 論点ディスカッション（軽量版）＋理解の照返し |
| `/github-planning`・`/gitlab-planning` の起票前ゲート | AC ウォークスルー（全 L2 Issue）＋未決事項の依存チェック |
| `/dev-tasks`「実装前設計」 | 設計オプション比較＋AC ウォークスルー（着手時の照返し）＋ no-op の照返し確認（大原則） |
| `/dev-tasks`「実装（Green）」 | 曖昧さ規律（実装中に検出した曖昧さの確認） |
| すべての工程（横断） | 曖昧さ規律 |

## 関連

- AC の構造（L1/L2/L3・Given/When/Then・起票前ゲート・実装フロー）は `.claude/rules/spec-driven.md`。
- 設計書（「主要な設計判断」節・`status: draft` の先行作成・承認）は `.claude/rules/design-doc.md`。
- 要件定義書の節構成・生成手順は `/project-setup` スキル、起票の手順は `/github-planning`・`/gitlab-planning` スキル、開発フローのタスク敷設は `/dev-tasks` スキル。
- AC からのテスト変換（TDD）は `.claude/rules/tdd.md`。
