# コードレビュー規則

コードレビューを行う際は以下に従うこと。

## レビューしないこと

フォーマット・Lint・型エラーはコメントしないこと。ツール（Prettier/ESLint/Ruff/TypeScript コンパイラ等）が担当する。

> ただし**ビジュアル品質（レイアウト・配色・余白・タイポ・操作性・崩れ）はツールが担保できない**ため、レビュー対象とする（下記「フロントエンド/UX 観点」）。「Lint が通る＝見た目も OK」ではない。

## レビュー観点

- 要件充足
- 設計の複雑さ・重複ロジック
- エッジケース（null、空配列、境界値）
- セキュリティ（インジェクション、XSS、認証・認可漏れ）
- パフォーマンス（N+1 クエリ、不要な計算）
- **Core Web Vitals（Web/UI の体感性能を含む差分のみ）**: ヒーロー画像・初期描画に影響する変更は、`chrome-devtools-mcp` の `debug-optimize-lcp` スキル（内部で `performance_start_trace`）で性能トレースを取り LCP を TTFB／リソース読込遅延／読込時間／レンダリング遅延の 4 サブパートに分解して確認する（`lighthouse_audit` は a11y/SEO 向けで**性能は対象外**）。「Lint が通る＝速い」ではない。
- エラーハンドリング（握りつぶし・不適切なフォールバックがないか）
- コメント/ドキュメント正確性（コメントが実装と一致するか・古い/誤誘導コメント＝コメント腐敗が無いか・公開 API の docstring が実態を反映するか。詳細審査は `pr-review-toolkit:review-pr` が起動する comment-analyzer に委譲する）
- **設計書整合**: 変更が `docs/design/` の該当領域ファイルに正しく反映されているか（図〔`sequenceDiagram`／`erDiagram`〕の更新を含む。領域の取り違え・実質を伴わない更新で `commit-msg` の規律だけ通していないか。構造は `.claude/rules/design-doc.md`）
- **依存追加時のライセンスと既知脆弱性**: 新規依存のライセンス確認（コピーレフト混入・商用利用可否。納品要件は `.claude/rules/operations.md` の第三者ソフトウェア一覧）と既知脆弱性（npm audit／pip-audit 等の依存監査。深掘りは `security-reviewer` に委譲する）
- **カバレッジの著しい低下**（`commands.coverage` 設定時のみ計測。数値目標の強制ではなく劣化検知のシグナルとして報告する。`.claude/rules/testing-strategy.md`）
- **フロントエンド/UX 観点（UI 変更を含む差分のみ）**: アクセシビリティ（キーボード操作・フォーカス可視・コントラスト・代替テキスト・ARIA）／レスポンシブ（主要ブレークポイントで崩れない）／デザイントークン整合（色・余白・タイポをハードコードせず既存トークンを使う）／コンポーネント再利用（重複実装がないか）／ローディング・空・エラー状態の有無／実レンダリングのスクリーンショット確認。詳細な審査は `frontend-reviewer` エージェントに委譲する。

## コメント接頭辞（必ず使用すること）

| 接頭辞 | 意味 |
|--------|------|
| `must:` | 必須修正（マージ前に要修正） |
| `should:` | 強く推奨 |
| `nit:` | 軽微な指摘（任意） |
| `question:` | 質問（回答が必要） |
