# TDD（テスト駆動開発）規則

すべての `feature/*` / `fix/*` 実装は Red-Green-Refactor サイクルで進める。言語・テストフレームワークに依存しない原則をここに記す。具体的なテストフレームワーク・コマンドは **CLAUDE.md の「プロジェクト設定」節**を参照すること。

## Red-Green-Refactor サイクル

1. **Red** — Issue の受け入れ条件（AC）から失敗するテストを書く
   - AC を 1 つ選び、失敗するテストを書く
   - コンパイルエラーも Red とみなしてよい
   - 実装コードを書く前にテストが失敗することを確認する
   - 画面要素を伴う AC は、ロジックの単体テストに加えて**コンポーネントのレンダリング/操作テスト**（Testing Library 等）も Red の対象に含める（ビジュアル/レスポンシブ/a11y の目視確認は Green 後に `frontend-reviewer` で行う＝下記「原則」）

2. **Green** — テストが通る最小の実装を書く
   - そのテスト 1 つが通ることだけを目標にする
   - 設計の整理・重複排除は後回しにする

3. **Refactor** — テストを変えずにコードを整理する
   - 重複を排除し、命名を改善する
   - テストがすべて通ることを確認しながら行う

AC の数だけこのサイクルを繰り返す。1 件ずつ回し、すべての AC が通ったら PR/MR を作る。

## テストの命名

| 要素 | 形式 | 例 |
|------|------|---|
| テスト関数名 | `対象_条件_期待結果` | `generateApiKey_asAdmin_returnsPrefixedKey` |
| 表示名（あれば） | 日本語で「対象 — 条件 → 期待結果」 | `"generateApiKey — 管理者は接頭辞付きキーを受け取る"` |

テスト本体には `// given / // when / // then`（または各言語のコメント慣習）で 3 ブロックを明示する。

---

## AC → テスト変換パターン

Issue の AC:
```
- [ ] Given 有効な API キーを持つユーザー, When 一覧APIを呼ぶ, Then 一覧が返る（200）
```

対応するテスト（疑似コード。実際の記法は各言語のフレームワークに従う）:
```
test "findAll — 有効な API キーで呼ぶと一覧が返る":
    # given
    repository.findAll() を 1 件返すようスタブする
    # when
    result = service.findAll()
    # then
    result の件数が 1 であることをアサートする
```

UI コンポーネントを伴う AC の例（描画も検証対象にする）:
```
AC: - [ ] Given ログインフォーム, When 無効なメールで送信, Then role=alert でエラーが表示される

test "LoginForm — 無効なメールで送信するとエラーが表示される":
    # given
    LoginForm を render する
    # when
    メール欄に "bad" を入力し送信ボタンを click する
    # then
    getByRole("alert") にエラーメッセージが表示されることをアサートする
```

## 原則（言語非依存）

- **AC 1 件 = テスト 1 つ（以上）**。AC に対応しないテストや、AC に対応するテストの欠落を作らない。
- **アサーションは表現力の高いものを優先**（例: JS/TS は `expect`、Java は AssertJ、Python は `assert` + pytest）。等値比較だけに頼らず、意図が読めるアサートを書く。
- **テーブル駆動**で複数入力をまとめる（パラメータ化テスト機構が各言語にあれば使う）。
- **ユニットテストは外部 I/O に依存させない**。重い統合（実 DB・実ブラウザ・実サブプロセス）が必要なケースは統合テストとして分離する。分離した統合テストの実行は `commands.integrationTest`（配置と実施タイミングは `.claude/rules/testing-strategy.md` の結合テスト規約）。
- **フロント変更時は「単体テスト緑」で完了としない**。実ブラウザを要するビジュアル/レスポンシブ/アクセシビリティの確認は単体から分離するが、**分離＝放置にしない** — Green 後に `playwright`／`chrome-devtools-mcp` で実描画スクリーンショットを取得し、`frontend-reviewer` でコントラスト・キーボード操作・状態表示・崩れを確認する（`/dev-tasks` の「ビジュアル/UX 確認」タスク）。テストが緑でも見た目が酷い状態を見逃さない。
- **モックの限界を理解する**。値オブジェクト/レコード型などモックできない型は実インスタンスを生成する。
- `feat`/`fix` コミットには対応するテストを必ず含める（テストなしコミット禁止）。

## 言語別のテスト規約

フレームワーク固有のパターン（テストクラス構造・モックライブラリ・実行コマンド・既知の落とし穴）は、プロジェクトごとに以下へ記述する：

- 実行コマンド → `CLAUDE.md` の「プロジェクト設定」節（`commands.test`）
- 具体的なコード規約 → プロジェクトの `test-writer` エージェント or 専用ルールファイル（`/project-setup` の `claude-automation-recommender`・`claude-md-improver` で生成・蓄積する）

---

## 単体より上のテストレベル

本ファイルが定めるのは**単体テスト**（TDD）の規則。単体より上のテストレベル — **結合・総合（システム）・受入（UAT）・性能** — の体系（定義・実施タイミング・テストデータ/環境管理・記録の置き場）は `.claude/rules/testing-strategy.md` を参照する。
