# ヒアリング充足ループ

> 対象フロー: 新規のみ ｜ 種別: 【人間ゲート】

`AskUserQuestion` は **1 回の呼び出しで最大 4 問**（各 2〜4 択）。この 1 回 4 問上限は維持しつつ、聴取は**固定ターン数で打ち切らず、要件が「充足」するまで反復する**。各ラウンドの後に充足判定し、**未充足の項目だけ**を次ラウンドで深掘りする。**これは 1 役割 = 1 タスク。内部反復であってタスク分割はしない。**

**コアラウンド（1 ラウンド目｜最大 4 問・依存順序のため最初に固定配置）**：何を作るか（目的）／提供形態（**Web / CLI / desktop / library**）／プログラミング言語／フレームワーク。
この 4 つは後続（LSP 行の再確認・設定ファイル確認・雛形選定）が依存するため、**必ず最初のラウンドに置く**。VCS ホストやプロジェクト名などの後続値は後のラウンドで良い。

#### 充足判定とループ

各ラウンドの後、次のチェックリストで**未充足の項目**を洗い出す（言語・FW 非依存の汎用項目。`.claude/rules/spec-driven.md` の 4 節スケルトンと概念的に重なる）。

- **必須（充足ゲート — これが埋まるまで反復）**
  - 要件定義書（「要件定義書の作成」タスク）の必須項目：**背景・目的（想定ユーザー／アクター。権限ロール含む）／スコープ／主要機能／業務フロー（主要ユースケース・業務の流れ＝アクターと操作の順序。業務システム・Web では必須、それ以外は「該当なし」の明示でも可）／非機能要件（対象環境・セキュリティに加え、定量目標〔応答時間・同時利用者数・データ規模〕・データ保護〔個人情報の有無・保持/削除・暗号化〕・バックアップ/DR〔RPO/RTO〕・監視方針・〔深掘り観点〕i18n/タイムゾーン。いずれも「標準／該当なし」の明示でも可）／テスト方針（「標準＝TDD＋結合は分離」の明示でも可）／技術選定／リリース・運用（デプロイ先・提供形態・データ移行の有無。「該当なし」の明示でも可。デプロイ先の確定値は「プロファイル確定」タスクの `commands.deploy`・デプロイ CI 雛形の有効化に使う）**。
  - 後続が依存する確定値：**提供形態（Web/CLI/desktop/library）／言語／フレームワーク（「なし」の明示も可）／パッケージマネージャ・`checks` 候補の方向性**（ここで確定するのは「編集後チェックを行うか／対象言語の方向性」まで。`tsc`・`ruff` 等の具体ツール名を聴取で固定列挙はしない。具体コマンドは「雛形構築」タスクのツール選定と「プロファイル確定」タスクで確定）。
  - **（提供形態が Web の場合のみ）UI 要求**：**ターゲットユーザーと利用文脈／デザインの方向性（トーン・参照デザイン/競合・既存ブランド資産の有無）／主要画面の構成／デザインシステム・コンポーネントライブラリの採否**。コアラウンドで提供形態が Web と確定したら、次の深掘りラウンドから必須集合に加える（他の必須と同様に前進保証〔停滞→安全なデフォルト適用〕の対象。非 Web では対象外）。これが無いと「どういう見た目を目指すか」が要件定義書（「要件定義書の作成」タスクの UI/UX 方針節）→ Issue の視覚 AC → `frontend-reviewer` の照合基準まで一度も捕捉されず、フロントエンドの出来が事後レビューでも判定不能になる。
- **任意（脱出・デフォルト可）**：プロジェクト名（ディレクトリ名から推定可）／VCS ホスト（「バージョン管理・リモート同期」タスクで remote から確定可）／`checks` 候補の細部（`/setup-*`・recommender が補完）／その他の要望。

> **「充足」= 各必須項目が、「雛形構築」タスクと「プロファイル確定」タスクに足る具体度で埋まっている状態**。網羅性ではなく具体度を見る。明示的に「特になし／標準」と確認できた項目はそれで充足とみなす。スタック固有の固定質問リストは持たない（上の項目は言語・FW 非依存）。

**ループの回し方（終了条件を正しく守らせる）**

1. **未充足の必須項目だけ**を、次の `AskUserQuestion` ラウンド（引き続き 1 回最大 4 問）で深掘りする。**充足済みの項目は再質問しない**（毎回ゼロからやり直さない）。
2. **数値上限（ラウンド数の天井）は設けない**。終了条件は次の 2 つだけ：
   - **(T1) 自動終了**：必須項目がすべて充足した（充足ゲート通過）。
   - **(T2) 利用者終了**：利用者が全体ゲート（後述）で「これで十分／残りはデフォルトで」を選んだ。
3. **無限ループは「前進保証」で防ぐ（提示で終わらせず必ず適用する）**：ある必須項目が**停滞**した（深掘りしても具体化しない＝同一項目が **2 ラウンド連続で充足しない**）ら、同じ問いを回し続けない。その項目の**安全なデフォルト適用を 1 回提示**し、それでも利用者が具体値を選ばなければ、**Claude がそのデフォルトを適用して未充足の必須集合から除外する**（適用した値と影響は明示する。安全なデフォルトが無い核心項目は「未確定のまま進む＝雛形/プロファイルが不完全」と影響明示した上で集合から除外する）。「提示するだけ」「利用者の脱出に委ねるだけ」では集合は減らないため、**脱出が選ばれなくても適用する**。これで各停滞項目には必ず「充足」か「適用デフォルト化」のどちらかが起き、未充足の必須集合は毎ラウンド**単調に減る** → 数値上限がなくても必ず停止する。

**脱出肢は 2 系統で常時提供する**（**深掘りラウンド**＝コアラウンド以外のすべてのラウンド）

- **全体ゲート（＝脱出 T2 の入口）**：**2 ラウンド目以降は例外なく**、その深掘りラウンドの**冒頭**に残りの未充足項目を提示する専用の `AskUserQuestion` を置き、`[深掘りする]／[これで十分・残りはデフォルトで進める]` を選ばせる（データ収集前のコアラウンドには置かない）。ここで後者を選ぶことだけが「脱出（T2）＝聴取全体の終了」にあたる。
- **項目デフォルト（個別スキップ。脱出ではない）**：深掘り各設問の選択肢に「デフォルトでよい／不明」を含める。**どちらを選んでも、その項目はデフォルト適用＝未充足集合から除外して前進させる**（「不明」＝利用者が値を持たないケースとして Claude がデフォルトを当てる）。これは個別項目のスキップであって聴取全体の終了ではない（残りの必須が充足／適用デフォルト化されるまでループは継続）。

**脱出（T2）を選んだ時、または全必須が充足／適用デフォルト化で確定した時**は、**未確定だった項目と適用したデフォルト（およびその影響）を明示**してから「要件ディスカッション」タスクへ進む（適用したデフォルトは要件定義書の「未決事項」節に記録される＝「要件ディスカッション」「要件定義書の作成」の両タスク）。必須項目をデフォルトで埋めて進む場合は「雛形・プロファイルが不完全になりうる」旨を伝える（最終判断は利用者に委ねる）。

**末尾で LSP/プラグイン行を再確認する**：言語が確定したので、「環境バリデーション」タスクで「保留」にした「LSP / プラグイン」行を再確認する（使用言語の LSP プラグインが `.claude/settings.json` で有効か。足したら `/plugin` 再読込（または再起動）を挟む。`.claude/settings.json` を編集するなら **auto mode はオフ**）。LSP が不要／後回しなら利用者に明示判断させる。

`【完了条件】必須チェックリスト（背景・目的〔アクター含む〕／スコープ／主要機能／業務フロー／非機能〔定量目標・データ保護・バックアップ/DR・監視・i18n/タイムゾーン〕／テスト方針／技術選定／リリース・運用〔デプロイ先・データ移行〕／提供形態・言語・FW・PM 方向性／（Web のみ）UI 要求＝ターゲットユーザー・デザイン方向性・主要画面・デザインシステム採否）が全て「充足」または「適用デフォルト化」になり、未充足必須が 0 件であること、および LSP 行の再確認結果を提示。`
