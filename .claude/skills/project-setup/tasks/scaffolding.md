# 雛形構築・設定ファイル・後始末

> 対象フロー: 新規のみ ｜ 種別: auto

- 必要なライブラリ・フレームワークをインストールする（パッケージマネージャはヒアリング結果に従う）。
- 提供形態に応じたディレクトリ構成を作成する（例: `frontend/` + `backend/`、`src/` + `tests/` など）。
- Lint/フォーマットは `/setup-js`・`/setup-python` 等を活用する。
- **提供形態が Web（フロントを含む）の場合は「デザインの土台」も立てる**（言語・FW 非依存の原則。具体ライブラリはスタック依存）:
  - (a) デザイントークン（色・余白・タイポの定義）の置き場を用意する（デザインシステム/コンポーネントライブラリの採否は**「ヒアリング充足ループ」タスクの UI 要求で聴取済みの値を使う**〔重複聴取しない〕。採用しない場合は最小のトークン定義ファイルを置く）。トークンを JSON 等のデータに集約しておくと (e) のトークン可視化が使える。
  - (b) **【必須】コンポーネントカタログを立てる** — コンポーネントを**状態別（通常/ローディング/空/エラー/各バリアント）に隔離して見られる**もの。**既定実装は Storybook**。component 非対応スタック（サーバレンダリング/テンプレート系等で story が成立しない場合）は同等のプレビュー手段（コンポーネント/パーシャルのプレビューページ等）に読み替える。それも成立しないなら **opt-out 理由を設計書（`docs/design/` の UI 領域）に記録**する。
  - (c) **【必須】a11y lint** を `/setup-js`（`eslint-plugin-jsx-a11y` 等）で含めさせる。
  - (d) **【必須】初回スクリーンショット**を `playwright`／`chrome-devtools-mcp` で撮り、以後の `frontend-reviewer` の基線とする（スクショは UI 変更時の DoD でも必須）。
  - (e) **（任意・提案）トークン可視化**: トークンを JSON 等に集約していれば、設計書を `.mdx` 化して色チップ/余白バー/文字見本として描画できる（手順は `.claude/rules/design-doc.md` の「トークン可視化」。フレームワーク非依存＝データ→HTML で、実コンポーネント埋め込みのようなビルド脆弱性が無い）。
  - (f) **【必須】コンポーネントテスト環境** — コンポーネントの描画/操作テストが走る環境（DOM 環境。JS なら jsdom 等）と、**最小の描画テスト 1 本が緑**であること。(b) カタログと同じ「**既定（DOM テスト環境＋Testing Library 等）＋非対応スタックは同等手段に読み替え＋成立しないなら opt-out 理由を設計書〔`docs/design/` の UI 領域〕に記録**」パターン。これにより `.claude/rules/tdd.md` が要求する「画面要素 AC のレンダリング/操作テスト」を土台から書ける（DOM 環境が無いまま描画テストを書いて落ちる事故を防ぐ）。具体ライブラリはスタック依存（`/setup-js` や recommender が補う）。
- **「プロファイル確定」タスクで `checks` に登録する予定のコマンドが参照する設定ファイルを、ここで必ず生成しておく**。型チェック・Lint・ビルドの各コマンドは、対応する設定ファイルが無いと編集時チェックが失敗する（または意図したチェックが走らない）。`checks` に入れる各コマンドについて「それが解決する設定ファイルが揃っているか」を確認する。
  - これは言語・フレームワーク非依存の一般原則。**具体的にどのファイルが要るかはスタックごとに異なる**ため、このテンプレートは固定の一覧を持たない（フレームワーク公式の scaffold が生成するか、`/setup-js`・`/setup-python` や recommender が補う）。迷う場合は、そのチェックコマンドを実際に 1 回実行し、設定ファイル不足のエラーが出ないことを確かめる。
- **スキャフォールドツールの後始末（言語・FW 非依存の一般原則）**：雛形生成ツール（`create-next-app`・`vite`・`uv init`・`cargo new` 等）は副作用を残すことがある。**「バージョン管理・リモート同期」タスクの `git add` 前に**解消しておく。具体的にどのファイル/ディレクトリかはスタックごとに異なるため固定一覧は持たない（迷う場合は生成物を確認する）。
  - **入れ子の `.git`**：既存リポジトリ外で実行すると、サブディレクトリに `.git` を自動生成する（入れ子リポジトリ）。後の `git add` で gitlink 化／失敗の原因になる。各ツールに「git を作らない」オプションがあれば付けて予防し、できてしまった分はここで片付ける：
    - **検出**：**Glob ツールで `**/.git/HEAD`** を検索する（`find` はシェル差—特に Windows—で挙動が変わるため、クロスプラットフォームな Glob を使う）。`**/` はルート直下も含むため、結果にはリポジトリルートの `.git`（あれば）も混ざる点に注意。
    - **除去**：検出されたうち **サブディレクトリ配下の `.git` だけ**を一覧化し（**ルート直下の `.git` は除外**）、ユーザーに提示して**承認を得てから** Claude が Bash で `rm -r <その .git ディレクトリ>` を実行する。プラットフォーム別コマンドは案内せず Bash の `rm -r` に統一する。出荷時 `.claude/settings.json` の deny に `Bash(rm -rf *)` があるため **`-f` は付けない**（`rm -r` は実行時に承認プロンプトが出る＝破壊的操作の明示確認として適切）。
  - **スタブ／サンプルファイル**：雛形は仮のエントリポイントやサンプルを置くことがある。意図した構成に合わないものは削除し、重複したエントリポイントを残さない。
  - **ビルド成果物の無視設定**：ツールが生成する成果物ディレクトリが `.gitignore` で除外されているか確認する。ルートに無いものはルート `.gitignore` に言語/FW に応じて追記する（テンプレートのルート `.gitignore` はフレームワーク固有の成果物を持たない方針）。ただし**ロックファイル（`package-lock.json`／`uv.lock`／`Cargo.lock` 等）は成果物ではない＝`.gitignore` に入れずコミットする**（成果物と誤分類して無視しない。規律の正は `.claude/rules/git-workflow.md`「依存ロックファイル」）。

ここで判明した値（言語・パッケージマネージャ・ビルド/テスト/開発サーバ起動（dev）/Lint/フォーマット/（分離して持つなら）結合テスト・カバレッジ・デプロイの各コマンド・`frontendDir`〔ルート直下にフロントを展開したなら `"."`〕・生成した設定ファイルと対応する `checks` 候補コマンド等）は控えておく（**実際のプロファイル書き込みは「プロファイル確定」タスクで確定する**）。

`【完了条件】予定 checks 候補コマンドが設定ファイル不足エラー無しで 1 回実行できる実行ログを提示し、かつ意図しない入れ子 .git・重複エントリポイント・未 ignore 成果物が残っていないことを生成物確認で示した。**さらにアプリのランタイム依存が実際に解決していることを実走で示す**＝判明した（予定の）`test`／`build` コマンド（無ければ主要依存を 1 つ `import` するだけの最小スモーク）が**依存解決エラー・未解決 import 無しで 1 回完走したログ**を提示する（Lint・型チェックの緑は依存解決を保証しない＝例: pyproject の `[project]` 欠落で `uv sync` がランタイム依存を入れないのに `ruff check` は緑、という偽陽性を防ぐ）。提供形態が Web の場合はさらに「デザインの土台」の個別証拠＝(a) トークン定義のパス・(b) カタログ起動ログ（または opt-out の設計書記録）・(c) a11y lint 実行ログ・(d) 基線スクショ（docs/screenshots/baseline-*.png）のパス・(f) 最小の描画テストが緑の実行ログ（または opt-out の設計書記録）、を提示する（【必須】を宣言だけで completed にしない）。`
