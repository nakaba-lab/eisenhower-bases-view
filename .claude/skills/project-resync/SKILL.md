適合済みプロジェクトの**スタックが大きく変わった節目**で、自動化（hooks/サブエージェント/スキル/プラグイン/MCP）の推奨・CLAUDE.md・プロジェクトプロファイルを**現行スタックへ再同期**する対話型フロー。あわせて**要件定義書と現状の乖離・依存の健全性（鮮度・既知脆弱性）**も点検する。初回適用（`/project-setup`）の後にだけ意味を持つ。各工程でユーザーの確認を取りながら進め、破壊的操作はしない。

> **保守ノート（管理側向け）**: 下記タスク 1〜4 は `/project-setup` の「推奨キュレーション＋適用」「CLAUDE.md 改善キュレーション」「テンプレ自己言及の最終検証」「プロファイル確定」の各タスクと同じ契約（recommender 適用 → improver 減算 → 自己言及 grep → プロファイル2層同期）。完全スタンドアロンで重複保持しているため、**一方を変えたら他方も見直す**こと。recommender/improver の呼び出し名と AskUserQuestion ゲートの型（options 構成・multiSelect 運用）は両者で一致させる（header 文言はコンテキストに合わせて差し替えてよい）。タスク 5（要件定義書の整合点検）・タスク 6（依存の鮮度・脆弱性点検）は resync 固有で、`/project-setup` 側に対応タスクは無い。

## 進行管理（重要）

ステップの飛ばし・順序乱れを防ぐため、`/project-setup` と同じく **Task システム**で各工程を **1 タスク = 1 役割**に分解し、直列タスク化して管理する。

### ロックの大原則

- 各タスクは **1 役割**。複数の作業を 1 タスクに束ねない。
- 各タスクの `description` 末尾に必ず次の 2 行を置く:
  ```
  【完了条件】<観測可能な単一の証拠：コマンド出力／生成ファイルパス／差分／AskUserQuestion の回答>
  【証拠なしに completed 禁止】上の証拠を提示する前に TaskUpdate で completed にしない。提示できないなら in_progress のまま留め、不足を報告する。
  ```
- **【人間ゲート】** 印のタスクは、`AskUserQuestion`／利用者の承認を得るまで `completed` にしない。隣接タスクを 1 ターンに詰め込まない。

### 線形チェーン不変条件

- 各タスクは `TaskUpdate` の `addBlockedBy` で**直前タスク 1 本だけ**に繋ぐ（単一の線形鎖）。
- タスク生成直後に `TaskList` を読み、`[先頭] → … → [末尾]` の単一鎖になっており、**役割名と件数（6 件）が想定どおり**であることを名前で 1 つずつ照合する。
- 照合が通って初めて先頭タスクを `in_progress` にし、Step 0 完了とする。

> Step 0（適合済み判定）はタスク集合を決める前段ゲートのため、タスク化せず先に確定する。

## Step 0: 適合済み判定（前段ゲート）

`.claude/project-profile.json` を読み、**初回適用が済んでいるか**を確認する。

- `project-profile.json` が無い、または `name` が空（`""`）など**未適合の状態**なら、再同期は意味を持たない。「このプロジェクトはまだテンプレート適合（初回セットアップ）が済んでいません。先に `/project-setup` を実行してください」と案内して**終了する**（タスクを生成しない）。
- 適合済み（`name` 等が埋まっている）なら、下記 6 タスクを生成する。

### タスク生成（6 タスク）

`TaskCreate` で以下を順に作成する（`description` には呼ぶスキル・確認事項・末尾のロック 2 行を含める。詳細手順は「各タスクの手順」を参照）。**`TaskCreate` の subject はタスク表の subject を一字一句使い、`description` には表のセル（単一責務・種別）を逐語転記する（要約・言い換え・省略をしない）**。表が長い場合も切り詰めない — 完了条件・人間ゲート表記が落ちると進行の歯止めが消える。

| # | subject | 単一責務（要点） | 種別 |
|---|---------|------------------|------|
| 1 | 推奨キュレーション＋適用 | recommender 実行 → 提示 → AskUserQuestion 採否 → 承認分のみ `.claude/` 適用（既存と重複適用しない） | 【人間ゲート】 |
| 2 | CLAUDE.md 改善キュレーション（減算） | improver 実行 → 除外指定のみ問う → 残集合で自己編集 | 【人間ゲート】 |
| 3 | テンプレ自己言及の最終検証 | grep で TEMPLATE-ONBOARDING・自己言及文の残存を確認（通常 no-op の安全網） | auto |
| 4 | プロファイル再調整（2層同時更新） | profile と CLAUDE.md「プロジェクト設定」節を新スタックへ差分調整 | 【人間ゲート】 |
| 5 | 要件定義書の整合点検（乖離・必須節） | `docs/要件定義書.md` と現状の乖離（スコープ・主要機能・非機能）と必須節の欠落を点検し、乖離は承認のうえ改訂＋「変更履歴」節へ追記 | 【人間ゲート】 |
| 6 | 依存の鮮度・脆弱性点検 | パッケージマネージャの監査コマンド（`npm audit`／`pip-audit` 等）で既知脆弱性・依存の鮮度を点検し、結果と対処（Issue 起票・更新自動化）を提示 | auto |

`blockedBy` 連鎖（完全直列・ヘッド = 推奨キュレーション＋適用）:
```
CLAUDE.md 改善キュレーション   ← blockedBy 推奨キュレーション＋適用
テンプレ自己言及の最終検証     ← blockedBy CLAUDE.md 改善キュレーション
プロファイル再調整           ← blockedBy テンプレ自己言及の最終検証
要件定義書の整合点検         ← blockedBy プロファイル再調整
依存の鮮度・脆弱性点検       ← blockedBy 要件定義書の整合点検
```

全タスクの生成と `blockedBy` 連鎖が完了し、線形チェーン照合を通過し、最初のタスクを `in_progress` にして初めて Step 0 完了とする。

## 各タスクの手順

各小節 = 1 タスクの実行詳細。**サブスキルは Skill ツールで呼ぶ**（`plugin:skill` 形式の名前を `skill` 引数に渡す）。**Agent ツール（サブエージェント）では呼べない**（同名のエージェントは存在せず `Agent type ... not found` になる）。

### 1. 推奨キュレーション＋適用【人間ゲート】

> このタスクは `.claude/` 配下に書き込むため、**auto mode はオフ**にしておく（auto mode では `.claude/` 配下の書き込みがブロックされる）。

1. `Skill("claude-code-setup:claude-automation-recommender")` を実行する。**読み取り専用**スキルで、現行コードベースを解析して推奨を出すだけ・ファイルは変更しない。
   - **kind が web（フロントを含む）の場合は、recommender に「a11y lint・ビジュアル回帰・機能 E2E（ブラウザ操作）スモーク/ランナー（`playwright` 等）・デザインシステム/コンポーネントカタログ・スクリーンショット確認フック」を提案観点に含めるよう促す**（project-setup の「推奨キュレーション＋適用」タスクと同じ Web シード。言語・FW 非依存は維持しつつ視覚品質の観点を足す）。
   - **kind を問わず、次の観点も recommender の提案に含めるよう促す**：デプロイ自動化・環境定義／エラートラッキング・ヘルスチェック・ログ整備／性能・負荷テストランナー（要件定義書の非機能定量目標の計測手段）／依存の脆弱性監査・更新自動化（`npm audit`・`pip-audit`・Dependabot・Renovate 等）／OSS ライセンス監査／テストデータ管理（フィクスチャ・シード・個人情報のマスキング）／API を公開・消費する場合のスキーマ駆動（OpenAPI 等）と契約検証（project-setup の「推奨キュレーション＋適用」タスクと同じ共通シード。具体ツールは recommender がスタックに合わせて選ぶ）。
2. 推奨を**種別別（hooks / subagents / skills / plugins / MCP）に件数つきで提示**する。**既に `.claude/` に存在する自動化は「既存」と明示**し、新規（net-new）と区別する。
3. **キュレーションゲート（`AskUserQuestion`）**:
   - header: `再同期の適用`
   - question:「recommender が現行スタックに対し N 件の自動化を提案しました（hooks ◯ / サブエージェント ◯ / スキル ◯ / プラグイン ◯ / MCP ◯。うち既存 ◯）。読み取り専用の提案で、まだ何も適用していません。`.claude/` への適用方針を選んでください。」
   - options:「全て承認（推奨）／一部を除外（適用しない項目を次の設問で選ぶ）／個別に確認（1 件ずつ採否）／追加要望あり」
   - **「一部を除外」時のみ** 2 問目を multiSelect（除外する種別 → 種別内項目）。選択肢は recommender の実出力から動的生成する。
4. **承認サブセットのうち net-new のみ** Claude 自身が適用する：承認 hooks/permissions → `.claude/settings.json`、承認サブエージェント → `.claude/agents/<name>.md`、承認スキル → `.claude/skills/<name>/SKILL.md`、承認 MCP → `.mcp.json`。**既存と同名のものは重複作成しない**（更新が要るなら可否を確認）。net-new の承認スキルを `.claude/skills/<name>/SKILL.md` に作る際は、`skill-creator:skill-creator` で起草→evals→発火精度（description）の最適化まで回す（project-setup の推奨適用と同じ導線。recommender は提案のみで作成・最適化はしない）。
5. **profile 値（`commands`・`checks`・`vcsHost` 等）はここで書かない**（タスク 4 に集約）。適用後、何を入れ何を除外したかのサマリを提示する。

`【完了条件】推奨ごとの適用/除外判断（AskUserQuestion の回答）と、承認分（net-new）のみを反映した適用/除外サマリを提示。【証拠なしに completed 禁止】回答取得前に .claude/ へ適用しない・回答を得る前に completed にしない。`

### 2. CLAUDE.md 改善キュレーション（減算）【人間ゲート】

1. `Skill("claude-md-management:claude-md-improver")` を実行する。品質レポートを出した後、**承認を得て CLAUDE.md を自身で編集する**スキル（自前の承認機構を持つ）。現行コードベースと CLAUDE.md のズレを検出・改善する。
2. **減算ゲート（`AskUserQuestion`）**：improver 自身の承認と二重化させないため、テンプレ側ゲートは**「除外したい個別提案の指定」だけ**を問う。
   - header: `CLAUDE.md 再同期`
   - question:「improver が CLAUDE.md の改善案 M 件を提示しました（追加◯ / 修正◯ / 削除◯）。除外したい改善案を選んでください（複数選択可・未選択＝全件を適用）。残った案を improver が CLAUDE.md に反映します。」
   - options:**M ≤ 4** は各提案を multiSelect（除外する項目）／**M ≥ 5** は除外する種別（追加案 / 修正案 / 削除案）→ 種別内 multiSelect。選択肢は improver の実出力から動的生成。
   - **保持対象（プロジェクト設定節・標準ワークフロー/各規則の説明・Hooks/プラグイン説明・コーディング規約）に触れる削除案は、除外候補としてハイライト**して過剰除去を防ぐ。
3. 除外集合を確定 → improver に「除外を除いた集合で CLAUDE.md を編集」させる（improver が自分で書く。Claude が代筆しない）。編集対象は **CLAUDE.md 系のみ**（`settings.json`・agents・hooks・skills は触らない＝タスク 1 の担当）。CLAUDE.md は `.claude/` 外なので auto mode の制約は受けない。

`【完了条件】improver の提案が利用者の除外指定を反映して CLAUDE.md に適用され、レポートが放置になっていないことを提示。【証拠なしに completed 禁止】除外指定の回答を得る前に CLAUDE.md を確定編集しない・completed にしない。`

### 3. テンプレ自己言及の最終検証【auto】

improver（タスク 2）の編集で、テンプレ自己言及（`TEMPLATE-ONBOARDING` マーカ／「これは汎用テンプレートである」「まず最初に `/project-setup` を実行」等のセットアップ前提の自己言及文）が CLAUDE.md 本文へ復活していないかを grep スイープで最終確認する。適合済みプロジェクトでは通常 no-op の安全網。

- grep 相当で次が CLAUDE.md に残っていないか検査する：`TEMPLATE-ONBOARDING` マーカトークン／セットアップ前提の自己言及文。
- 見つかった場合は当該箇所を除去（またはプロジェクト固有の説明へ書き換え）し、再度 grep して残っていないことを確認する。
  - **除去対象**：テンプレートそのものへの自己言及・セットアップ前提の導入文。
  - **保持対象**：「プロジェクト設定」節、標準開発ワークフロー・各規則の説明、Hooks・プラグイン説明、コーディング規約など、適合後も有用な恒久情報（過剰除去しない）。

`【完了条件】grep 相当で CLAUDE.md に TEMPLATE-ONBOARDING マーカおよびセットアップ前提の自己言及文が 1 つも残っていないことを提示（残存があれば除去後の再 grep 結果も提示）。`

### 4. プロファイル再調整（2層同時更新）【人間ゲート】

> `.claude/project-profile.json` に書き込むため、**auto mode はオフ**にしておく。

現行スタックに合わせて **`.claude/project-profile.json`** と **CLAUDE.md「プロジェクト設定」節**を**差分調整**する（2 層がずれないよう必ず同時更新）。**初回適用と違い greenfield ではない**——既存の有効値は保持し、変わった/増えた項目だけを反映する。

1. 現行コードベースから判明した値（言語・パッケージマネージャ・各コマンド・`frontendDir` 等）と、現在の profile を突き合わせ、**変更/追加が必要な項目を差分として洗い出す**。
   - **kind=web の場合は「デザインの土台」の有無も点検する**: デザイントークン定義／コンポーネントカタログ（Storybook 等）／a11y lint（`eslint-plugin-jsx-a11y` 等）／基線スクリーンショット（`docs/screenshots/baseline-*.png`）。欠けているものは差分として提示し、**承認を得て導入まで実行する**（提案止まりにしない。カタログは AskUserQuestion で採否、a11y lint は `/setup-js` 等（既存 ESLint・フレームワーク管理 lint があれば `/setup-js` が検出して `AskUserQuestion` で非破壊に確認する）、基線スクショは `commands.dev` 起動→`playwright` で取得。project-setup の「デザインの土台」タスクと同じ内容の事後補完）。
2. 対象キー: `name`（通常不変）/ `kind` / `languages` / `packageManager` / `commands.{build,test,dev,lint,format,migration,integrationTest,coverage,deploy}` / `frontendDir`（UI のあるディレクトリ。ルート直下なら `"."`、UI を持たないなら `"none"`。**kind=web で空のまま残さない**）/ `checks`。
   - 言語が増えた場合は `checks` に言語ごとのエントリを追加（例: TypeScript → `{ "match": "**/*.ts", "command": "npx tsc --noEmit", "cwdFromRoot": true }`、Python → `{ "match": "**/*.py", "command": "uvx ruff check", "cwdFromRoot": true }`）。
   - **`match` は編集ファイルの絶対パスに対する Python `fnmatch`**。`fnmatch` は **globstar 非対応**で `**` は実質 `*` と同じ（パス区切り `/` も含めて何にでも一致）。先頭は `**/` か `*` で始める（絶対パスは `/…` や `C:/…` で始まるため。`src/**/*.ts` のように具体パスで始めると絶対パスに一致せず**チェックが沈黙する**＝無一致）。検証: `python3 -c "import fnmatch; print(fnmatch.fnmatch('/abs/path/to/app.ts', '**/*.ts'))"`（`True` なら一致）。
   - 各コマンドが参照する設定ファイルが揃っていることを確認してから `checks` に入れる（無いとチェックが失敗する）。
3. **差分を提示して承認を得る（`AskUserQuestion`）**:
   - header: `プロファイル再調整`
   - question:「現行スタックとの差分はこの通りです（変更前→変更後）。この内容で project-profile.json と CLAUDE.md『プロジェクト設定』節を更新してよいですか？」
   - options（**multiSelect は使わない**）:「更新する（推奨）／修正する（指摘箇所を直して再提示）」
4. 承認後、project-profile.json と CLAUDE.md「プロジェクト設定」節を**同じ値で同時更新**する。

`【完了条件】project-profile.json と CLAUDE.md「プロジェクト設定」節の値が完全一致し、checks が空でないなら各 match を絶対パス fnmatch で検証済みであることと、変更前後の差分・AskUserQuestion の承認を提示。【証拠なしに completed 禁止】承認を得る前に確定書き込みしない・completed にしない。`

### 5. 要件定義書の整合点検（乖離・必須節）【人間ゲート】

`docs/要件定義書.md` は「初期合意」だが、スタックが変わる節目では**現状と乖離**していることが多い。放置すると要件→Issue のトレース（`.claude/rules/spec-driven.md`）・テスト方針・非機能の定量目標（総合テストの基準＝`.claude/rules/testing-strategy.md`）が古い前提のまま使われるため、ここで点検する。

1. **存在確認**：`docs/要件定義書.md` が無い（または opt-out 済み＝CLAUDE.md に記録あり）場合は、その旨を提示して完了する（必要になったら `/project-setup` の既存フロー「要件定義書の有無確認」で生成できる旨を案内）。
2. **乖離点検**：スコープ・主要機能・非機能要件（定量目標・データ保護・バックアップ/DR・監視方針）を現状の実装・運用と突き合わせ、乖離を一覧化する。
3. **必須節の欠落点検**：必須節（背景・目的〔想定ユーザー／アクター〕／スコープ／主要機能／業務フロー／非機能要件／テスト方針／技術選定／リリース・運用／未決事項／変更履歴。Web は UI/UX 方針も）の欠落を確認する（「標準／該当なし」の明示があれば充足扱い。リモートでも `design-doc-check` CI の requirements-doc-check ジョブが欠落を警告する）。
4. **改訂の承認（`AskUserQuestion`）**：乖離・欠落の一覧と改訂案を提示し、承認を得てから該当節を改訂する。改訂したら**「変更履歴」節に 1 行追記**する（日付・変更内容・合意者）。乖離・欠落が無ければ「点検済み・改訂なし」を証拠として提示する。

`【完了条件】乖離・欠落の点検一覧と、改訂した場合は差分＋「変更履歴」節への追記＋AskUserQuestion の承認（改訂が無ければ「点検済み・改訂なし」の根拠、要件定義書が無ければ opt-out／未作成の確認結果）を提示。【証拠なしに completed 禁止】承認を得る前に確定編集しない・completed にしない。`

### 6. 依存の鮮度・脆弱性点検【auto】

依存の放置（大きく古いバージョン・既知脆弱性）はスタック変更の節目で顕在化しやすい。パッケージマネージャに応じた監査を 1 回実行し、結果を提示する。

1. **監査の実行**：profile の `commands`・`checks` に監査系コマンドが定義されていればそれを実行する。無ければパッケージマネージャ標準の監査コマンド（例: `npm audit`／`pip-audit`。スタックに応じて選ぶ）を 1 回実行する（PATH に無い・監査機構を持たないスタックは、その旨と導入候補を提示して完了する）。
2. **結果の整理と対処の提案**：既知脆弱性（特に重大度の高いもの）と大きく古い依存を一覧化し、対処を提案する——修正 Issue の起票（`/github-planning`・`/gitlab-planning`）／更新の自動化（Dependabot・Renovate 等。未導入ならタスク 1 の共通シード「依存の脆弱性監査・更新自動化」で提案済みかを確認）。
3. ここでは**点検と提案まで**とし、依存の一括更新は行わない（更新は通常の Issue 駆動フローで行う＝破壊的操作をしない原則）。

`【完了条件】監査コマンドの実行ログ（または実行不能の理由と導入案内）と、検出結果・対処提案の一覧を提示。`
